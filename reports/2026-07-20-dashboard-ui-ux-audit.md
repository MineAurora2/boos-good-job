# 仪表盘 UI/UX 审计报告

- 审计日期：2026-07-20
- 审计范围：`dashboard/index.html`、`dashboard/styles.css`、`dashboard/app.js` 当前 `main` 代码；未运行写入型操作，未修改生产代码。
- 判定原则：将“数据缺失”“请求失败”“指令已提交/已生效”和“筛选后的统计”分别视为不同用户状态；所有结论均以以下行号为依据。

## 现状优势

1. **控制面已有回执链路。** 实例状态被拆为连接、执行、同步三轴（`dashboard/app.js:1698-1718`、`1743-1748`），生命周期操作会显示提交、接收、应用、超时、失败等反馈（`dashboard/app.js:2223-2237`），而不是只用一个“运行中”标签。
2. **计划配置的渐进展开和输入反馈基础可靠。** 开关用 `aria-controls`、`aria-expanded` 与 `inert` 约束折叠区（`dashboard/index.html:130-140`）；日期/时间选择器在关闭时恢复触发点焦点（`dashboard/app.js:1386-1391`、`1469-1486`），时长滑块兼具 pointer 与键盘操作（`dashboard/index.html:205-223`、`dashboard/app.js:2168-2184`）。
3. **窄屏不是简单缩放。** 920/680/430px 分别重排侧栏、控制区、定时编排、日志和表格（`dashboard/styles.css:922-978`）；数据加载也会保留本地筛选、选中项与刷新提示（`dashboard/app.js:2485-2495`）。
4. **可访问性已有起点。** 主导航、关键指标、日志、图表、地图及多个按钮有标签或 live region（`dashboard/index.html:24-25`、`81`、`122-125`、`240-252`、`274`、`312-319`）；删除和结束全部均为显式确认操作（`dashboard/index.html:521-545`）。

## 主要问题

### P0：错误被伪装成“空数据”

**证据。** `loadData` 请求异常后，在记录为空时直接写入 `{ summary: {}, deliveries: [] }` 并调用完整渲染（`dashboard/app.js:2485-2495`）；表格空态固定解释为“没有找到匹配记录，试试调整时间范围或筛选条件”（`dashboard/index.html:398-405`）。这会把首次请求失败、认证取消、服务断开和真实零记录呈现为同一视觉结果。

**影响。** 用户会误调筛选或误判投递量为零，运营人员尤其难以判断是否应重试、检查连接或联系管理员。

**建议。** 建立 `loading / ready / filtered-empty / source-empty / error / unauthorized` 状态机。错误态应保留失败原因、上次成功更新时间及“重试”按钮；只有 `ready + records.length === 0` 才使用业务空态。认证取消要显示“未授权，数据未加载”，不能借用零记录 UI。

### P0：认证取消会抑制后续认证，且取消原因不可见

**证据。** 取消会把 `authPromptDismissed` 设为 `true`（`dashboard/app.js:226-229`），后续 `requestAuthToken` 直接解析空字符串（`dashboard/app.js:231-245`）；收到 401 后 `authorizedFetch` 直接返回该响应（`dashboard/app.js:256-266`），最终进入上述伪空数据分支（`dashboard/app.js:2488-2495`）。

**影响。** 用户一次误按取消后，本页会静默失去重新认证机会；刷新、轮询、管理区请求都难以形成可恢复的操作路径。

**建议。** 取消仅结束当前请求，不应作为会话级永久禁止标志；将“重新验证”暴露在错误横幅和顶部连接状态中。明确区分 401、网络失败、服务端 5xx，并为每种状态提供一次可预期的重试动作。

### P1：单页承担六类高频任务，认知与滚动成本过高

**证据。** 固定侧栏将总览、脚本监控、分析、地图、记录、系统管理都定位到同一文档锚点（`dashboard/index.html:24-44`）；同页连续放入实时控制与定时编排（`108-232`）、分析地图和记录表（`265-410`）、配置、LLM、简历、提示词管理（`412-503`）。导航点击仅 `scrollIntoView`（`dashboard/app.js:3366`）。

**影响。** “今天是否正常投递”“调整定时计划”“编辑密钥”有不同风险和心智模型，却共享一个长页面与全局刷新，关键操作易被淹没；移动端的长页面成本更高。

**建议。** 将 IA 切为“运营总览 / 自动化控制 / 投递记录 / 分析报表 / 系统管理”五个路由或主视图。总览只保留可行动异常和核心 KPI；管理作为独立受限工作区。保留 URL/历史记录，以便直接回到某一运营任务。

### P1：状态含义仍有歧义，尤其是全局按钮与“期望/实际”关系

**证据。** HTML 初始三颗全局按钮都带 `.active`，却全部 `aria-pressed="false"`（`dashboard/index.html:123-126`）；运行时用“所有实例的期望状态是否 running”计算全局状态（`dashboard/app.js:1904-1914`），而实例卡同时显示期望状态、执行状态、同步状态（`dashboard/app.js:2223-2240`）。单实例操作按钮还会把 `starting/applying` 归入“已运行”而只提供“结束”（`dashboard/app.js:1732-1740`）。

**影响。** 视觉“active”、切换按钮的 `aria-pressed`、后端期望状态和实际浏览器状态没有一一对应；在离线、应用中、失败时，用户不容易判断命令是否真正生效。

**建议。** 统一用“命令目标 / 当前执行 / 同步结果”三行文案和颜色，不把命令按钮当状态标签。全局区域显示“X/Y 已生效、Y 个待回执、Z 个失败”，只在明确选择模式时使用 `aria-pressed`。为 applying 状态保留“查看回执/重试”，避免暗示已经运行。

### P1：账号上限编辑存在异步覆盖与重复卡片竞态

**证据。** 同一账号的输入会被所有匹配控件同步写值（`dashboard/app.js:3493-3501`）；保存前用账号 ID 和原始字符串记入草稿/待处理并立即重绘（`3481-3485`），完成后仅在 pending 值仍相等时结算（`1757-1763`、`3486-3488`）。控制数据又会每 3 秒轮询并重绘（`dashboard/app.js:3529-3537`）。

**影响。** 用户在请求返回前继续编辑、或同账号在多实例卡重复出现时，刷新可能使编辑焦点、草稿或服务端新值相互覆盖；“保存成功”不等于用户当前看到的值已持久化。

**建议。** 按 `accountId + revision` 保存本地草稿，编辑期间冻结该字段的轮询回填；保存使用乐观并发版本或请求序号，响应只可提交最新请求。将账号配额编辑集中为单一抽屉/行内编辑器，避免同账号多卡编辑入口。

### P1：弹窗和抽屉没有完整的焦点生命周期

**证据。** 删除、结束全部、认证弹层声明了 `aria-modal`（`dashboard/index.html:521-558`），但事件只绑定取消、遮罩点击和全局 Escape（`dashboard/app.js:3333-3340`、`3507-3513`）；没有看到焦点圈定、打开前触发元素记录或关闭后恢复。详情抽屉仅有 `aria-hidden`（`dashboard/index.html:510-519`），相应打开函数只填充内容（`dashboard/app.js:1490-1504`）。

**影响。** 键盘用户可 Tab 到被遮罩的背景控件，关闭后焦点可能丢失到文档开头；读屏用户难以稳定地知道当前活动上下文。

**建议。** 统一 `openDialog/closeDialog`：保存 opener、将焦点置于标题或首个可操作项、Tab 循环、关闭后恢复 opener，并在打开时使背景 `inert`。抽屉应具备 `role="dialog"`、`aria-modal`（或明确非模态语义）和同样的焦点策略。

### P1：移动端虽然重排，但操作密度和可达性仍不足

**证据。** 680px 下顶栏保留时间范围、主题、刷新、导出，导出变为仅图标的 36px 按钮（`dashboard/styles.css:954-956`、`973`）；430px 下仍把配额编辑压到一个数字框和两个 52/58px 操作列（`dashboard/styles.css:977-978`）；表格仅隐藏列（`dashboard/styles.css:967`、`979`），而不是提供卡片摘要/详情路径。侧栏开关只切 class，不更新 `aria-expanded` 或处理菜单焦点（`dashboard/app.js:3327`）。

**影响。** 小屏用户要在密集工具栏和长表格中横向/纵向切换；菜单状态也不能被辅助技术可靠感知。

**建议。** 移动端优先保留“异常、执行、确认”三种任务：将低频主题/导出移入更多菜单，记录表改为可展开摘要卡，配额编辑采用整行表单。移动菜单应有 `aria-expanded`、关闭按钮、焦点返回和背景遮罩。

### P2：字号与非文本对比度降低扫读能力

**证据。** 多处运营关键信息使用 8-10px：监控说明和控制按钮为 8px（`dashboard/styles.css:189`、`207-209`），日志为 9px（`198-199`），侧栏状态为 9-10px（`127-134`），卡片补充说明也为 9px（`166`）。深色主题中还用 #4e6673、#48606d、#718896 等低饱和文本色（`dashboard/styles.css:113-134`、`189-199`）。

**影响。** 高分辨率大屏、缩放和低对比显示器下，运行日志、状态说明和按钮标签难以扫读；状态颜色也需要同时依赖图标/文本。

**建议。** 正文字号最低 12px，关键状态/操作最低 14px；将说明文字提高到可测量的 WCAG AA 对比度，并为成功/警告/失败同时提供文字、图标和形状。以 100%、200% 缩放及亮/暗主题分别做对比度验收。

### P2：键盘和 ARIA 实现不完整或语义不一致

**证据。** 时间范围分段按钮没有 `aria-pressed`，仅依赖 class（`dashboard/index.html:69-74`，对比计划周期已使用 `aria-pressed` 的 `147-151`）；管理标签只更新 `aria-selected`，未见 `aria-controls`、`id`、tabpanel 绑定（`dashboard/index.html:417-423`、`dashboard/app.js:3367-3371`）；列设置菜单由普通按钮切换 `hidden`，未见 `aria-expanded`（`dashboard/index.html:390-395`、`dashboard/app.js:3357-3361`）。地图虽支持 Enter/Space 和方向键（`dashboard/app.js:1009-1024`），是可复用的良好模式。

**建议。** 为所有分段控件采用一致的 button-group 或 tabs 模式；标签页补齐 `id`、`aria-controls`、`role="tabpanel"`，支持左右箭头/Home/End；菜单补齐展开状态与 Escape 焦点返回。将地图的键盘处理抽为通用交互验收基线。

### P2：漏斗口径混合“当前筛选投递”与“全量审计”，易造成错误决策

**证据。** 漏斗将 `records.length` 作为“进入投递”，把 `state.payload.summary.evaluatedJobs` 作为评估量并直接相除（`dashboard/app.js:608-617`）；自身提示已承认投递按当前筛选、评估/阈值/异常来自全量审计（`617`）。总览每次筛选会调用该渲染函数（`dashboard/app.js:1502-1505`）。

**影响。** 选择“今日/城市/岗位”后，分子改变而分母不一定改变，转化率不再对应同一个人群或时间窗，可能把筛选效应误读为投递效率变化。

**建议。** 默认将全漏斗按同一筛选器、同一时间窗聚合；若服务端暂不支持，明确拆成“当前筛选投递数”和“全量评估转化”两张指标卡，禁止把它们画进同一比率。接口返回 `denominatorScope`、`generatedAt`、缺失原因和不可比标识。

## 实施优先级

1. **P0，先保障事实可信。** 在数据、认证、控制接口周围建立显式状态机；拆除“失败即空数组”；为认证取消提供再次验证。先补请求失败、401、空数据三组自动化/端到端测试。
2. **P1，随后降低运营风险。** 重构控制面状态模型和账号上限并发策略；统一所有模态/抽屉焦点管理；将管理区从长页拆出。对控制命令做延迟、离线、回执失败、连续编辑的场景测试。
3. **P2，最后提高效率与可访问性。** 提升字号/对比度，统一分段控件与 tabs/menu 语义，完善移动记录视图，校正漏斗分母。以键盘、读屏、200% 缩放、320px 和 1440px 视口做回归验收。
