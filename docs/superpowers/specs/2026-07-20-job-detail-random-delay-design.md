# 职位详情随机延迟设计

## 目标

为获取职位详情增加独立的随机延迟区间，覆盖搜索页和聊天页的全部详情入口，降低连续详情访问形成固定节奏的概率，同时不改变现有投递随机延迟。

## 配置模型

在 `frontend` 下新增两个字段：

- `detailRandomDelayMinMs`：获取职位详情前随机延迟下限，默认 `0`。
- `detailRandomDelayMaxMs`：获取职位详情前随机延迟上限，默认 `0`。

两个字段允许 `0` 到 `600000` 毫秒，上限不得小于下限。旧配置通过默认配置合并自动补齐这两个字段；默认值为零，因此升级后不会引入额外等待。

详情随机延迟与现有投递随机延迟使用不同区间，但共同受 `antiDetectionEnabled` 总开关约束。总开关关闭或详情延迟区间为零时，详情访问保持当前即时行为。

## 浏览器脚本

保留 `antiDetection.delay()` 的现有投递延迟语义，增加 `antiDetection.detailDelay()`。两者复用同一个可中断分片等待实现，但分别读取投递区间和详情区间，避免配置相互影响。

详情等待插入以下三个访问边界：

1. 搜索页 `getJobInfo()` 写入详情时间戳并打开详情窗口之前。
2. 聊天页首次处理未沟通过职位时，点击职位卡片之前。
3. 聊天页收到新消息且尚未发送简历时，再次点击职位卡片之前。

等待发生在详情访问动作之前，因此不计入 `detailTimeout`。搜索页暂停或停止时取消当前详情访问；聊天页暂停或停止时跳过当前分支，不再点击职位卡片。停止信号仍由现有生命周期守卫中断。

## 配置界面与文档

Dashboard 的通用配置表单会自动展示新增字段，只需补充中文标签。`user_config.example.json` 和 `readme.md` 同步记录默认值、范围、总开关关系及三个入口均受影响的语义。

## 错误处理与兼容

- 非数字、布尔值、负数或超过 `600000` 的值由现有配置校验拒绝。
- `detailRandomDelayMaxMs < detailRandomDelayMinMs` 时拒绝保存并返回明确错误。
- 浏览器拦截详情窗口、详情回传超时及投递流程的既有错误处理保持不变。
- 现有 `randomDelayMinMs/randomDelayMaxMs` 继续只控制投递前后等待。

## 测试

- Python 配置测试覆盖默认值、合法边界、越界值和上下限倒置。
- Node 用户脚本测试覆盖详情区间的闭区间取值、总开关关闭、零区间、分片等待和暂停中断。
- 源码契约测试确认搜索页一个入口和聊天页两个入口均在打开或点击详情之前调用 `detailDelay()`。
- 运行项目完整测试脚本，包含 Python 单元测试、JavaScript 语法检查和 Node 契约测试。
