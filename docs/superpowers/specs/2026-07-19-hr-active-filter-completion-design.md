# HR 活跃筛选完整状态与严格白名单设计

## 背景与目标

当前系统只识别 6 种 HR 活跃状态。运行日志中实际出现的 `2周内活跃`、`2月内活跃`、`3月内活跃`、`4月内活跃`、`5月内活跃`、`近半年活跃` 和 `半年前活跃` 都会落入 `unknown`，而旧逻辑对 `unknown` 无条件放行，导致已开启的筛选被绕过。

本次修改补齐日志中已经确认的全部活跃时间文案，并把筛选统一为严格白名单：只有勾选的状态可以通过；原始文案为空或无法识别时归入“未知”，且只有勾选“未知”才通过。

## 状态模型

状态按活跃时间由近到远排列，最后放置无法判断时间的“未知”。

| 配置值 | 页面文案 |
| --- | --- |
| `online` | 当前在线 |
| `just_now` | 刚刚活跃 |
| `today` | 今日活跃 |
| `within_3_days` | 3 日内活跃 |
| `this_week` | 本周活跃 |
| `within_2_weeks` | 2 周内活跃 |
| `this_month` | 本月活跃 |
| `within_2_months` | 2 月内活跃 |
| `within_3_months` | 3 月内活跃 |
| `within_4_months` | 4 月内活跃 |
| `within_5_months` | 5 月内活跃 |
| `within_half_year` | 近半年活跃 |
| `half_year_ago` | 半年前活跃 |
| `unknown` | 未知 |

不增加日志中从未出现的“昨日活跃”“1 周内活跃”“1 月内活跃”“6 月内活跃”或“1 年前活跃”等推测字段。未来出现的新文案按“未知”处理，避免未经确认的时间归类。

## 配置与兼容

后端配置枚举、默认值和管理页多选项使用同一组 14 个状态。默认配置选择全部状态，保持首次使用时不过滤的行为；已有 `user_config.json` 的选择不自动扩充，新状态默认保持未选中，避免升级后意外放行。

旧版 `hrActiveMinLevel` 继续按由近到远的时间顺序迁移成白名单。`unknown` 只能用于新的多选配置，不接受为旧版最低档位。

配置保存继续拒绝空数组、重复值和未知配置值。浏览器端若收到“筛选已开启但没有任何有效选项”的异常配置，则所有岗位均不通过，避免错误配置静默关闭筛选。

启动加载遇到显式 `null`、非法类型、全非法数组或无法迁移的旧最低档位时，将其归一化为空白名单而不是回退到默认全选；完全没有提供该字段时才使用默认配置。

## 数据流与过滤行为

1. 管理页保存 HR 活跃多选值，后端校验后写入配置。
2. 浏览器脚本启动时获取最新前端配置。
3. 岗位详情页原始 HR 文案经过空白归一化和精确规则映射；已有同义文案 `在线`、`刚刚在线`、`刚活跃`、`3天内活跃` 继续兼容。
4. 关闭筛选时所有状态通过；开启筛选时仅当归一化状态存在于所选白名单中才通过。
5. 未通过的岗位继续记录 `job_hr_filtered`，日志同时保留原始文案、归一化状态和当前白名单，便于后续发现新文案。

管理页、监控卡片、详情和导出使用相同的状态标签映射，避免新状态退化为代码值或“未知”。

## 测试与验收

- 配置测试覆盖 14 项默认值、任意多选、`unknown` 合法选择、空值/重复值/非法值拒绝，以及旧最低档位迁移。
- 管理页契约测试覆盖 14 个选项及固定顺序。
- 浏览器脚本测试覆盖每个真实文案和已有同义文案的归一化。
- 过滤测试覆盖：关闭筛选全部放行、精确多选、未勾选“未知”时拒绝未知、勾选后放行未知、开启筛选但有效选项为空时拒绝。
- 回归验证同时覆盖搜索页与聊天页共享的过滤函数，并运行项目现有 Python 与 Node 测试集。

## 变更范围

预计修改 `app/config.py`、`app/storage/admin_store.py`、`dashboard/app.js`、`web_script.js`、`readme.md`、`user_config.example.json` 及对应本地测试。不会修改当前用户已经保存的 `user_config.json`，也不涉及岗位评分、AI 筛选或投递流程的其他规则。
