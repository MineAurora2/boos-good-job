# 浏览器会话手动启动与心跳限流设计

## 目标

1. 每次油猴脚本创建新会话并接入后端时，都保持 `stopped`，不得继承上一个会话的 `running` 状态。
2. 页面刷新、站内跳转和浏览器重新打开都属于新会话，必须由 Dashboard 或油猴悬浮按钮再次手动开启。
3. 常规运行心跳从 5 秒调整为 15 秒。
4. 首次接入和控制状态回执仍可立即触发心跳，但任意两个实际心跳请求之间至少间隔 2 秒；间隔内的重复请求合并为一次。

## 根因

- 后端会持久化每个 worker 的 `desiredState`。油猴脚本生成新 `sessionId/sessionEpoch` 后，首次心跳仍会收到旧会话留下的 `running`，随后自动创建执行器并运行。
- 油猴目前有 5 秒周期心跳，同时焦点、联网、页面显示、日志及控制状态变化都会直接请求心跳。请求进行中只做一次尾随合并，但请求完成后可立即再次发送，因此忙碌阶段仍会出现密集心跳。

## 方案

### 新会话强制停止

后端 `RuntimeMonitor.heartbeat()` 在接受控制协议客户端的心跳后，比较持久化 worker 会话和当前心跳会话。首次登记保持现有 `stopped` 默认值；已登记 worker 接入新的 `sessionId/sessionEpoch` 时，在返回控制状态之前原子执行以下更新：

- `desiredState` 设为 `stopped`；
- 分配新的全局 revision 和 operationId，避免客户端把旧控制误判为已处理；
- 清除旧 `controlAck`；
- 保存新的会话字段并持久化；
- 唤醒控制长轮询等待者。

同一会话的后续心跳不得再次重置目标状态。这样 Dashboard、持久化状态和油猴收到的控制目标始终一致。即使上一个会话仍在运行，新会话也会成为较新的有效会话；旧会话的后续心跳按现有会话新鲜度规则拒绝。

### 心跳统一调度

`ControlAgent` 使用一个统一的心跳调度入口：

- 启动时立即请求一次心跳；
- 周期定时器调整为 15 秒；
- 控制状态、页面可见性、网络恢复及运行遥测触发仍调用同一入口；
- 调度器记录上一次实际请求开始时间；不足 2 秒时不立即发送，而是只保留一个延迟任务；
- 延迟期间新增请求继续合并，不额外创建定时器；
- 当前请求执行中收到触发时，保留一次尾随请求，但同样遵守 2 秒最小间隔。

控制长轮询保持现状。它负责快速接收控制指令，并继续刷新后端临时在线时间；心跳负责上传完整遥测、日志和控制回执。

## 错误与生命周期

- 首次心跳失败时仍按统一调度重试，不绕过 2 秒最小间隔。
- 鉴权失败、过期会话和断网安全暂停保持现有处理。
- 新会话重置状态的持久化失败沿用后端现有原子写入错误路径，不返回与磁盘状态不一致的成功结果。
- 油猴脚本停止时清理周期定时器和延迟心跳定时器，避免页面卸载后残留调度。

## 测试

### 后端

- 已登记且目标为 `running` 的 worker 使用新会话接入后返回 `stopped`。
- 新会话重置会更新 revision、operationId、会话字段并清除旧回执。
- 同一会话的后续心跳保留当前目标状态，不重复重置。
- 首次登记仍默认 `stopped`。
- 旧会话在新会话接入后仍被拒绝。

### 油猴脚本

- 周期定时器使用 15 秒。
- 首次启动立即发出心跳。
- 2 秒内多个心跳触发只产生一个延迟请求。
- 请求进行中收到多次触发只产生一个受限流约束的尾随请求。
- 超过 2 秒的触发可立即发送。
- 控制状态回执最终能够上传，不因合并而丢失。

## 不在范围内

- 不改变 Dashboard 控制按钮和 API 形态。
- 不改变 30 秒在线 TTL、20 秒控制长轮询或断连安全暂停阈值。
- 不恢复旧会话的运行进度；每个新会话都要求人工重新开始。
