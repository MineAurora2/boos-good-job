# 2026-07-19 投递转化复盘汇报设计

## 目标

面向管理层，用一份 16:9、7 页的 PPT 解释 2026-07-19 进入投递岗位偏少的原因，并纠正“投递转化率 3%”的统计口径。

## 结论口径

- 当日评估岗位：473 个（`job_decision_consumed`）。
- 当日进入投递：95 个（`delivery_state.db` 中 `claimed_at` 为 2026-07-19 的记录）；其中 89 个已发送、4 个仍排队、1 个预留、1 个异常。
- 当日真实进入投递率：`95 / 473 = 20.1%`；已发送率：`89 / 473 = 18.8%`。
- 面板显示 3% 的原因：`dashboard/app.js` 的 `renderFunnel` 使用全历史评估数 3,665 作为分母，计算 `95 / 3665 = 2.6%` 后显示为 3%，没有按今日筛选评估量联动。
- 对 473 个评估岗位按可审计事件拆分：HR 活跃筛除 191 个；AI 筛除 35 个（在 HR 通过后记录）；随机跳过 18 个；规则阈值筛除 2 个；进入队列事件 94 个；数据库另有 1 个预留记录；剩余 132 个没有明确的后续事件。
- 132 个只能标为“领取前未记录/中断”：代码存在重复岗位预检通过后直接跳过、预检服务不可用、随机延迟中断和控制停止等无统一审计事件，不能把它们全部断言为重复岗位。
- HR 筛选是最大可确认损耗：191 / 378 = 50.5% 的未进入投递岗位；这些事件均记录当时只允许 `online`、`just_now`、`today`，而岗位状态为 `within_3_days`、`this_week` 或 `this_month`。
- 运行侧影响：两个主账号分别进入 43、44 个；第三账号 22:01 后才启动，发生 23 次“浏览器拦截职位详情窗口”，仅进入 8 个。主要扫描时段集中在 02:54-04:57 和 17:40-22:28，中间存在约 12 小时 40 分空档。

## 叙事与页面

1. **封面 / 一句话结论**：用大数字展示“真实 20.1%，不是 3%”，副标题说明统计口径错位。
2. **指标纠偏**：并列展示错误公式与正确公式，注明全历史分母与同日分母。
3. **当日漏斗**：473 → 95 → 89，使用水平漏斗和状态标签，避免把“进入队列”和“已发送”混为一谈。
4. **损耗归因**：用条形图展示 191 / 132 / 35 / 18 / 2，并标注“132 为日志盲区，不是已证实的重复岗位”。
5. **运行效率**：账号分组柱状图（43、44、8）与运行时间轴，突出第三账号晚启动和弹窗拦截。
6. **原因定级**：将原因分为“已证实”“强推断”“需补日志”三层，避免过度归因。
7. **整改优先级**：按 P0/P1/P2 列出修正面板分母、调整 HR 活跃范围、增加领取前审计事件、提前启动并处理浏览器拦截。

## 视觉与实现

- 使用深色墨蓝背景、青绿色主色、橙色警示和红色异常，保证图表与正文高对比。
- 所有图形使用 PPT 原生形状和文本，不依赖外部图片；中文字体优先 Microsoft YaHei，英文与数字使用 Aptos。
- 每页保留右下角数据来源和口径说明；关键数字至少 28pt，正文 14-18pt。
- 生成脚本读取仓库当前 JSONL 与 SQLite 数据，固定使用 `2026-07-19` 的本地日期过滤，并输出一份数据快照 JSON，保证 PPT 可复算。

## 验证标准

- PPTX 能被 `python-pptx` 重新打开，页数为 7，所有页面都有标题和来源脚注。
- 数据快照中的核心数字与 SQL/JSONL 独立查询一致。
- 在没有 LibreOffice 的环境下，至少完成结构校验；如可用 PDF 渲染，再检查每页非空和无明显溢出。
